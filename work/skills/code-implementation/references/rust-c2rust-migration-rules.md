# Rust C2Rust Migration Rules

## Purpose

These rules apply to Rust code translated from C, especially code generated by C2Rust or similar migration tools.

Translated Rust is often unsafe, C-like, and non-idiomatic. The goal is not to rewrite everything at once.

The patch must preserve behavior first, then improve safety and maintainability incrementally.

## Priority Order

Use this order:

1. Preserve C behavior.
2. Keep Rust compiling.
3. Keep configured tests passing.
4. Reduce undefined behavior risk.
5. Shrink unsafe boundaries.
6. Improve idiomatic Rust locally.
7. Improve readability.

Do not reorder these priorities.

## Semantic Preservation

Before changing translated logic, check:

1. integer signedness;
2. overflow expectations;
3. pointer nullability;
4. pointer aliasing;
5. buffer length assumptions;
6. struct layout;
7. union usage;
8. global mutable state;
9. C string conventions;
10. allocation and ownership conventions.

## Unsafe Reduction

Allowed safe rewrites include:

1. replacing raw pointer iteration with slice iteration when length and validity are known;
2. replacing manual null checks with `Option<NonNull<T>>` when appropriate;
3. wrapping unsafe pointer logic in a small safe function with documented preconditions;
4. replacing C-style error codes with `Result` only when API boundary allows it;
5. replacing manual memory operations with safe container operations when ownership is clear.

Forbidden rewrites:

1. raw pointer to reference conversion without proving validity;
2. removing null checks without proof;
3. changing `repr(C)` structs used across FFI;
4. changing integer types without signedness and range analysis;
5. changing allocation ownership without release-path analysis;
6. replacing pointer arithmetic with slice access without length proof;
7. removing unsafe blocks without documenting why safety is now guaranteed.

## Layout and ABI

Preserve:

1. `#[repr(C)]`;
2. field order;
3. FFI-visible function signatures;
4. extern function ABI;
5. C-compatible enum layout when relevant.

Only change layout when the repair plan explicitly says the type is internal and not ABI-sensitive.

## C Strings

For C string handling:

1. distinguish `*const c_char` from Rust `String`;
2. validate null termination before conversion;
3. avoid assuming UTF-8 unless guaranteed;
4. prefer `CStr` for borrowed C strings;
5. prefer `CString` for owned C strings passed to C.

## Globals and Mutability

Translated C often uses globals or static mutable state.

Before changing global state:

1. identify initialization order;
2. identify threading assumptions;
3. identify aliasing assumptions;
4. identify whether state is externally visible;
5. prefer encapsulation over broad rewrite.

## Patch Summary Requirements

For C2Rust tasks, record:

1. original unsafe pattern;
2. replacement pattern;
3. why semantics are preserved;
4. unsafe blocks removed;
5. unsafe blocks retained;
6. ABI-sensitive items preserved;
7. tests or verification commands used.
