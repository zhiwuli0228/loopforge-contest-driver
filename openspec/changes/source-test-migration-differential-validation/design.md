## Context

Change 1 已提供真实 C 项目的源码、API、类型、调用关系和源测试清单；Change 2 已提供行为契约、状态转换和语义不变量；Change 3 已生成可构建 Rust 工程及实现映射。当前缺口是证明生成工程在测试层面不是“能编译但语义空心”，并且证明测试与差分验证框架本身不依赖 FlashDB 专属知识。

本变更处于完整 Rust 工程生成之后、编译与语义自愈闭环之前。它只能消费同一证据链中已通过门禁的输入，并输出可供 Change 5 定位修复的失败证据；不能通过删除测试、弱化断言或识别项目名来获得成功状态。

## Goals / Non-Goals

**Goals:**

- 将原始 C 测试场景、断言、fixture 和语义不变量迁移或等价覆盖为有效 Rust 测试。
- 使用 C 原实现作为 Oracle，对 Rust 重写执行同输入、同初始存储状态的差分验证。
- 覆盖返回值、错误、状态、遍历顺序、容量边界、持久化重开、失败恢复和 GC 前后可见状态。
- 通过固定随机种子的状态机测试和 mutation testing 证明测试能够识别关键语义错误。
- 建立项目无关门禁，禁止 FlashDB 定制、黄金答案、符号前缀分支和领域名特判。
- 输出完整、同一运行身份绑定且不可真空通过的测试与差分证据。

**Non-Goals:**

- 不生成或修复 Rust 核心实现；实现问题交给 Change 3 或 Change 5。
- 不执行编译器或差分错误驱动的自动修复闭环。
- 不执行正式干净 Linux E2E，也不声明最终 `READY_FOR_EVALUATION`。
- 不重新推断源码事实、行为契约或实现映射分母。

## Decisions

### 1. 源测试迁移以测试 IR 为中间层

迁移器先把源测试清单、断言、fixture、输入数据、期望输出、环境依赖和生命周期步骤转换为项目无关测试 IR，再从 IR 生成 Rust 测试模块。IR 中的实体只引用稳定源测试 ID、行为契约 ID、语义不变量 ID、API ID 和实现 ID，不保留项目名驱动的分支。

选择测试 IR 而不是直接从 C 测试文本渲染 Rust，是为了让覆盖统计、断言有效性扫描、差分向量生成和 mutation testing 共用同一分母，并让非 FlashDB 项目能够复用同一机制。

### 2. 覆盖率分母来自上游证据，不能由迁移结果决定

`source-test-map.json` 的分母来自 Change 1 的源测试和源断言清单，语义不变量分母来自 Change 2。每个源场景必须映射到迁移测试、差分向量或经证据批准的等价覆盖；每个源断言必须对应至少一个非恒真的 Rust 断言或结构化比较项。

不允许将无法迁移的测试从分母移除。无法建模的 fixture、平台依赖或不确定行为必须记录阻塞原因，并输出 `BLOCKED_WITH_REPORT`。

### 3. 差分 harness 只比较规范化观测值

C 原实现与 Rust 重写在隔离工作目录中获得相同输入、初始存储镜像、随机种子和故障注入计划。两侧执行后输出规范化观测值，包括返回码、错误类别、可见键值、时序记录、遍历序列、容量状态、持久化镜像摘要和恢复状态。

选择规范化观测而不是逐字节比较全部内部结构，是因为 Rust 迁移允许改变内部布局；但所有公开行为、持久化可见结果和契约规定的副作用必须一致。对于必须保持格式兼容的持久化内容，比较项必须明确记录字节级或结构级口径。

### 4. 状态机测试和 mutation testing 作为有效性证明

系统根据行为契约和状态转换生成固定随机种子的操作序列，并在 C/Rust 两侧执行。mutation testing 对生成 Rust 工程或验证适配层注入关键语义错误，例如错误返回码、删除未生效、容量边界 off-by-one、遍历顺序错乱、持久化重开丢失和恢复路径忽略损坏。

选择 mutation testing 是为了证明测试不是只执行路径或自比较。至少固定缺陷集必须被检测；若注入后测试仍通过，测试迁移门禁失败。

### 5. 反 FlashDB 定制是独立门禁

验证器扫描测试迁移器、差分 harness、生成测试、配置和报告模板，拒绝项目名、领域名、已知文件名、符号前缀、固定 API 名称列表、黄金输出和样例路径驱动的分支。允许这些字符串作为输入证据中的普通数据出现，但不能作为控制流条件、匹配规则或内置配置。

这条门禁必须在通用 fixture 和至少一个非 FlashDB fixture 上运行。若同一 schema 的非 FlashDB 项目不能通过测试迁移和差分框架的结构门禁，则本变更不能声称项目无关。

### 6. Cargo test 是必要门禁但不是充分证明

候选 Rust 测试执行 `cargo test --locked -- --nocapture` 并记录完整日志、测试数量和测试名称。成功状态还要求源测试映射、断言有效性、差分一致性、状态机一致性、mutation 检出率和反定制扫描全部通过。

只依赖 `cargo test` 退出码无法发现空测试、恒真断言或差分未执行，因此不能作为唯一成功依据。

## Risks / Trade-offs

- [C 测试依赖平台或时间行为] -> 将外部依赖抽象为可记录 fixture，不能稳定复现时记录阻塞原因，不从覆盖分母移除。
- [内部布局变化导致字节比较误报] -> 默认比较公开观测值，对持久化格式兼容要求使用契约标记的字节级比较。
- [mutation 注入破坏编译而非语义] -> 固定缺陷集区分编译型和语义型 mutation，报告必须记录检测来源，语义有效性不得只由编译失败证明。
- [反定制扫描误报输入证据中的项目字符串] -> 扫描区分只读证据数据和控制流/配置逻辑；误报必须有机器可读裁决，核心命中未裁决前阻塞。
- [差分执行成本高] -> 使用确定性向量分层运行，快速门禁覆盖源测试和关键状态机，完整门禁保留所有向量和 mutation 结果。

## Migration Plan

1. 固定源测试映射、测试 IR、差分向量、差分报告、语义不变量映射、mutation 报告和 cargo 测试日志 schema。
2. 实现同一证据链加载器，拒绝 Change 1/2/3 门禁未通过、run ID 混合、摘要不一致或工程内容陈旧。
3. 实现 C 测试到测试 IR、Rust 测试和 source-test-map 的生成路径，并加入断言有效性扫描。
4. 实现 C Oracle 与 Rust 重写的隔离执行、规范化观测、比较器和差分报告。
5. 实现固定种子状态机测试、固定 mutation 缺陷集和 mutation 检出门禁。
6. 实现反定制扫描和非 FlashDB fixture 结构验证，并接入 runner。
7. 只有全部测试与差分证据通过时才允许 Change 5 消费成功状态；失败时发布完整 `BLOCKED_WITH_REPORT`。

## Open Questions

- 实施时需固定 C Oracle 的构建方式和链接策略，避免把项目专属 build 脚本写入通用 harness。
- mutation 固定缺陷集的最小规模需要在实现前与运行时间预算一起确定，但必须覆盖返回码、状态修改、边界、遍历、持久化和恢复路径。
