## Context

现有 `c2rust_analysis.py` 主要通过文本模式提取函数和公开 API，无法可靠处理一般 C 项目中的宏、条件编译、复杂类型、函数指针和跨文件关系。现有证据也缺少与主分析器实现独立的集合核验。

该变更只建立可信源码事实层。输入是布局解析器确定的唯一 C 项目根目录且保持只读；输出是后续阶段可消费、可追溯、可 fail-closed 验证的 JSON 证据。

## Goals / Non-Goals

**Goals:**

- 完整发现当前输入的核心 C 源文件、公开 API、源测试、类型和跨文件关系。
- 所有分析策略从当前输入、布局解析和编译数据库派生，不内置题目身份或验收常量。
- 为每个结论保存文件、行号、符号及预处理配置等来源证据。
- 用独立实现扫描关键集合，防止主分析器遗漏后自证通过。
- 在零计数、核心解析失败、悬空符号或集合不一致时阻止后续代码生成。
- 生成稳定、版本化、可供机器验证的八类证据文件。

**Non-Goals:**

- 不在本 change 中定义项目专属行为契约、状态转换语义或 Rust 所有权映射。
- 不生成或修复 Rust 工程，也不迁移 C 测试。
- 不声明完整 C/Rust 语义等价或 `READY_FOR_EVALUATION`。
- 不修改、格式化或预处理写回 `SOURCE_ROOT` 中的任何文件。

## Decisions

### 1. 使用编译数据库驱动的预处理/AST 前端，保留受控回退

主分析器优先读取项目提供或构造的 `compile_commands.json`，逐翻译单元运行能够暴露预处理记录和 AST 的 C 前端。输出同时保留原始声明位置、定义位置和有效编译参数。缺少编译数据库时，可按项目配置生成明确的有限变体集合；无法覆盖核心文件的情况直接阻塞。

选择 AST 而非继续扩展正则，是因为函数指针、嵌套声明、宏展开和条件编译无法用文本模式可靠恢复。回退仅用于发现候选文件和构建编译参数，不能作为核心类型/API 完整性通过依据。

### 2. 采用规范化中间模型并从模型派生证据

分析结果先汇入带 schema version 的规范化模型：翻译单元、文件、符号、类型、声明/定义边、include 边、调用边、类型依赖、共享状态访问及预处理变体。稳定 ID 由规范化项目相对路径、符号限定名和声明类别产生；集合和边在写出前排序去重。

`source-inventory.json`、`public-api-map.json`、`type-map.json`、`call-graph.json`、`global-state-map.json` 和 `preprocessor-variants.json` 从同一模型派生，以避免多套提取逻辑造成内部漂移。每个核心实体必须至少带一个可定位的 source evidence。

### 3. 独立扫描器只核验关键集合，不复用主分析模型

独立扫描器使用不同的发现路径，直接遍历只读源码树并以轻量词法/预处理扫描提取核心 C 文件、公开头文件声明候选和测试文件。它不得导入主分析器的抽取函数或读取主分析结果来构造自己的集合。

验证器比较双方的核心源文件、公开 API 和源测试集合。独立扫描器可能产生候选超集，因此差异必须被分类为主分析遗漏、独立扫描误报或配置排除，并附证据；任何未解释差异均阻塞。

### 4. 明确定义核心范围和 fail-closed 门禁

项目根解析后，分析范围包含真实工程中的 `src`、`inc`、`tests` 以及配置声明的等价目录。生成文件、第三方依赖和示例是否属于核心范围必须由版本化配置明确表达，不能在计算覆盖率时静默排除。

验证器要求核心源文件、公开 API 和源测试计数均大于零；核心文件覆盖率、公开 API 声明/定义/证据覆盖率、核心类型成员建模覆盖率均为 100%；不存在未解释悬空符号；两种扫描的关键集合一致。任一条件失败时输出 `BLOCKED_WITH_REPORT` 和首个阻塞点，并禁止调用代码生成阶段。

### 5. 证据写入使用单次运行标识和原子发布

八类证据均包含同一 `run_id`、输入根标识、配置摘要、schema version 和生成状态。先写入运行临时目录，完成交叉引用与 schema 校验后再发布到当前运行的 trace 目录。验证器拒绝 run ID 不一致、路径不存在或来自旧运行的混合证据。

## Risks / Trade-offs

- [不同平台的 C 前端和编译参数导致 AST 差异] → 固定并记录解析器版本、目标三元组、宏和 include 参数；对路径和顺序做规范化。
- [条件编译组合爆炸] → 从实际构建配置和显式验收变体生成有限集合，并在证据中列出未覆盖变体；核心变体缺失时阻塞。
- [独立扫描器候选超集产生误报] → 支持带来源和原因的显式差异分类，但禁止无证据白名单。
- [大工程调用图分析耗时增加] → 按文件内容与编译参数摘要缓存翻译单元，同时每次重新执行全局集合核验。
- [解析器依赖增加环境配置成本] → 启动时做版本与能力预检，缺失依赖时给出确定的阻塞报告，不降级为正则成功。

## Migration Plan

1. 定义规范化模型、八类 JSON schema 和核心范围配置。
2. 将 AST 分析器接入现有分析阶段，在兼容模式下并行生成新旧证据以比较差异。
3. 实现独立扫描器和差异分类，增加通用 C fixture 的集成测试。
4. 将验证器切换为新 fail-closed 条件，并在门禁通过后才允许迁移规划或代码生成。
5. 删除只为旧证据结构保留的兼容路径；如需回退，仅回退门禁接线，保留新证据供诊断，不得将不完整分析标记为成功。

## Open Questions

- 实施时需根据 CI/Linux 环境现有工具链，在 libclang、Clang JSON AST 或 tree-sitter C 中确定主 AST 适配器；无论选择哪种，规范和 fail-closed 行为不变。
- 不允许固定特定项目验收版本；预处理配置来自编译数据库或调用方对当前输入的显式配置。
