## Context

Change 1 已提供输入 C 项目的完整源码事实，Change 2 已将其转换为公开 API 行为契约、状态转换、语义不变量以及 Rust 类型、所有权和端口规划。当前缺口是把这些已验证输入确定性地物化为完整 Cargo 工程，并独立证明核心函数、API、模块和调用边没有在生成过程中丢失或被占位实现替代。

本变更处于语义规划门禁和测试迁移之间。生成器只能消费属于同一证据链的已验证输入；输出必须能够 `cargo build --locked`，但构建成功不能替代实现完整性、模块关系、占位扫描和安全审计。

## Goals / Non-Goals

**Goals:**

- 生成包含迁移计划所列公共 API、状态模型和外部端口的完整 Cargo 工程。
- 禁止通过项目名、领域名、已知符号前缀或固定黄金答案选择生成逻辑。
- 为全部核心 C 函数、公开 API、模块和核心调用边建立到具体 Rust 符号与源码位置的可验证映射。
- 以 fail-closed 门禁拒绝不支持项、占位实现、伪逻辑、构建失败和超限或无说明的 `unsafe`。
- 支持目标模块及直接依赖范围内的确定性增量重新生成，并证明无关文件不变。
- 原子发布工程和生成证据，为 Change 4 提供稳定、可追溯的测试目标。

**Non-Goals:**

- 不迁移原始 C 测试，不执行 C/Rust 差分或 mutation testing。
- 不实现编译器或测试驱动的自动修复闭环；该能力属于 Change 5。
- 不执行正式干净 Linux E2E，也不声明 `READY_FOR_EVALUATION`。
- 不重新推断已由 Change 1、Change 2 确定的源码事实或业务语义。

## Decisions

### 1. 以验证后的语义 IR 为唯一生成输入

生成运行绑定源码分析 run ID、语义规划 run ID、输入摘要、schema 版本和工具版本。生成器从 `rust-migration-plan.json` 分配模块、类型、所有权和端口，从行为契约与状态转换实现函数逻辑，并用源码事实核对函数及调用边分母。

不采用直接从 C 文本再次生成 Rust 的旁路，因为它会绕过已验证的失败语义和所有权决策。任何输入缺失、门禁未通过、运行身份混合或摘要不一致都在写入当前工程前阻塞。

### 2. 分层生成 Cargo 工程并保留稳定实现身份

工程模块与端口仅从迁移计划的目标模块、类型映射、实现义务和边界依赖生成。每个生成符号具有稳定实现 ID，反向引用 C 函数/API ID、行为契约、目标模块和源码范围；模块合并、拆分或依赖反转沿用迁移计划中的等价理由。

选择稳定 ID 和结构化映射而不是仅扫描文件名，使验证器能够区分同名符号、检查具体实现位置，并为后续测试和修复精确定位目标。

### 3. 完整性验证独立于模板渲染结果

生成后发布 `implementation-map.json`、`unsupported-functions.json` 和 `module-edge-map.json`。验证器从上游分母重新计算核心函数、公开 API、核心模块和调用边集合，要求函数/API 映射率 100%、不支持项为零，并验证映射符号在 Rust 语法树中存在且具有非空可达实现。

验证器扫描 `todo!()`、`unimplemented!()`、`panic!()` 占位、空函数、与契约不符的恒定返回和未使用输入等伪实现模式。函数级生成失败不得中断主生成流程；生成器记录机器可读诊断，在模块和工程骨架完成后执行一次最终通用修复尝试。静态启发式命中可进入明确裁决流程，但核心命中未解决前不得通过；仅依赖 `cargo build` 的替代方案不能识别语义空壳。

### 4. 安全 Rust 优先并量化 unsafe 边界

核心状态由所有权和受控可变借用表达，平台能力经端口隔离。每处 `unsafe` 必须关联稳定 ID、源码位置、必要性、安全不变量和安全封装边界；比率按明确记录的 Rust 代码统计口径计算并严格小于 0.10。

完全禁止 `unsafe` 可能使必要的 FFI 或布局边界不可表达，因此保留有限例外；但缺失说明、统计分母为空或比例超限均阻塞。

### 5. 增量生成使用闭包清单和哈希事务

目标模块由模块依赖图计算允许变更闭包：目标本身及迁移计划明确的直接依赖。运行前记录整个生成树的规范化文件哈希，先在暂存目录生成并验证，再比较变化集合；闭包外任何文件变化都拒绝发布。相同输入的重复增量运行必须产生相同规范化内容。

选择暂存后原子替换而非原地覆写，以避免失败运行留下部分工程。完整生成也使用同一发布协议；回退时保留上一个已验证工程和失败报告。

### 6. 构建是必要门禁而非完整性代理

候选工程完成后在锁定依赖下执行 `cargo build --locked`，记录命令、工具链、退出码和完整日志。成功状态要求构建、实现映射、模块边、占位扫描和 unsafe 审计全部通过。任一失败都不得提前终止主流程；系统必须完成最终报告并把仍未解决的问题统一标记为 `BLOCKED_WITH_REPORT`。

## Risks / Trade-offs

- [复杂 C 控制流无法可靠模板化] → 先按已解析 IR 尝试通用修复与生成，例如字段赋值、输出指针、状态更新和已识别循环模式；函数级失败先记录诊断并允许其它模块继续生成，最终阶段统一修复后仍无法从源码事实、行为契约或状态转换证明等价的核心函数才进入不支持清单和最终报告，不生成猜测性占位。
- [占位检测误报合法常量函数] → 用行为契约、输入使用和源码证据联合裁决，并保存机器可读理由；不得静默忽略。
- [Rust 模块重构使一对一映射不自然] → 允许多对一或一对多映射，但要求稳定 ID、全部目标符号和等价理由完整。
- [unsafe 比率口径被操纵] → 固定统计工具、文件范围和分子分母定义，拒绝零分母，并逐处核对审计记录。
- [增量依赖范围不足导致工程不一致] → 从已验证模块边图计算直接依赖闭包，并在候选工程上重新执行完整生成门禁与构建。

## Migration Plan

1. 固定输入证据、生成输出和验证报告的 schema、稳定 ID、运行绑定与原子发布协议。
2. 实现 Cargo 骨架、模块、类型/状态、端口和函数体生成器，并建立多个不相关小型 C 项目的通用生成基线。
3. 实现独立映射验证、占位/伪实现扫描、unsafe 审计和锁定构建门禁。
4. 实现依赖闭包、暂存生成、文件哈希比较及增量发布/回退。
5. 接入 runner：只有完整生成门禁通过才允许 Change 4 消费工程；旧生成路径保留为回退时也不得绕过新门禁。

## Open Questions

- 实施时需固定 Rust edition、最低支持工具链和依赖版本；这些值必须进入生成运行元数据和 `Cargo.lock` 验证。
- `unsafe` 比率的最终计量单位需在实现前固定为语句、行或语法节点之一；无论选择何种口径，都必须确定、不可通过格式化稀释且严格执行 `< 0.10`。
